Upgrading from 1.25.1 to 1.26

Summary of Changes

Integration must be updated to reflect the following changes:

  • Removal of previously deprecated classes, functions, and properties, including removal of Legacy Capture. For more information, see Removed section below.
  • All screens associated with capture that are presented modally have been modified to to use UIModalPresentationStyle.overFullScreen instead of UIModalPresentatStyle.fullScreen.
  • SBSDKConfiguration and all its members previously conformed to an NSCoding protocol. The conformance has been removed, since the SDK is not archiving or unarchiving SBSDKConfiguration.

Removed

  • Legacy capture, deprecated in release 1.25 was removed. See the next section for the full list of removed items. To migrate to a new capture, please follow the Capture Changes guide in Upgrading from 1.24 to 1.25 section.
  • The start function inside PasswordAuthenticator was removed. Please follow The PasswordAuthenticator guide in Upgrading from 1.22 to 1.23 section to migrate to a new method.
  • The SBDataEventType was removed. Please follow the SBDataEvent Changes guide in Upgrading from 1.23 to 1.24 section to adjust your Receipt and Transaction observers to new methods.
  • Classes, functions, and properties deprecated in release 1.24 were removed. Review Deprecated list of Upgrading from 1.23 to 1.24 section for a full list of items removed.
  • Unused SensibillError codes deprecated in release 1.22 were removed. Review Deprecated list of Upgrading from 1.22 to 1.23 section for a full list of items removed.
  • Unused functions saveEmails and getAllEmails inside SBNonReceiptEmail were removed.

Deprecations

  • SensibillSDK class and all its functions were deprecated. Please see the following section on how to migrate.

SensibillSDK Deprecation

To migrate initWithAccessToken:
  1. Implement TokenProvider, as described in Custom Token Provider Authentication section. Inside the provideTokenReplacement function, construct a Credentials object by providing the accessToken and refreshToken you previously provided to the deprecated function:
class CustomTokenProvider: TokenProvider {
    func provideTokenReplacement(completion: @escaping (Credentials?, Error?) -> ()) {

        ...
        let credentials = Credentials.init(accessToken: accessToken, refreshToken: refreshToken)
    }
}
  1. Replace the deprecated method with the following code:
// Set user mode
Sensibill.shared.userMode = .single

// Use cacheIdentifier, which will be set automatically for single userMode.
let cacheIdentifier = Sensibill.shared.cacheIdentifier

// Create an instance of CustomRTokenProvider, created in step 1
let tokenProvider = CustomTokenProvider()

Sensibill.start(tokenProvider: tokenProvider, cacheIdentifier: cacheIdentifier) {

    guard case .none = error else {
        // error callback, similar to the one used previously
        errorCallback(error)
        return
    }

    // success callback, similar to the one used previously
    successCallback(true)
}
To migrate presentCapture functions:
  • The function SensibillSDK.presentCapture(inCurrentViewController: controller) can be replaced with ReceiptCaptureManager.shared.presentCapture(host: controller, expenseType: .unknown)
  • The function SensibillSDK.presentCapture(inCurrentViewController: controller, using: transaction) can be replaced with ReceiptCaptureManager.shared.presentCapture(host: controller, transaction: transaction)
  • The function SensibillSDK.presentCapture(inCurrentViewController: controller, expenseType: expenseType) can be replaced with ReceiptCaptureManager.shared.presentCapture(host: controller, expenseType: expenseType)
  • The function SensibillSDK.presentCapture(inCurrentViewController: controller, transaction: transaction, expenseType: expenseType) can be replaced with ReceiptCaptureManager.shared.presentCapture(host: controller, transaction: transaction, expenseType: expenseType)
To migrate UI helper functions:

The following functions were moved to other classes, their signature didn’t change:

  • The function displayReceipt was moved to SensibillReceiptViewController
  • The function pushToExportsController was moved to RecentExportsViewController. In Objective C function signature was changed from pushToExportsControllerFromNavigationController to pushToExportsControllerFrom
  • The function pushToSearchReceiptControllerFromController was moved to SBSearchReceiptsController and renamed to push (pushFromController in Objective-C)
To migrate utility callbacks:

  • Instead of checkIfServerReachableForConfigurationWithCompletionHandler, use NetworkReachabilityService.isAPIReachable, which provides an Error object in case of failure, in addition to a boolean result:

    NetworkReachabilityService.isAPIReachable { (isSuccessful, error) in

guard isSuccessful else {

    print("Failed to contact API. Error: \(error?.localizedDescription ?? "unknown")")
    return
}

print("Successfully contacted API")
}

  • Instead of registerAnalyticsTrackerListener, register your listener with SBAnalytics directly:

    extension MyClass: SBEventListenerInterface {

...
func register() {
    SBAnalytics.sharedInstance().addListener(self)
}
...
}

  • The registerErrorReceivingCallback is deprecated to be removed. Since this callback provided low-level networking errors, it didn’t accurately reflect the state of the SDK (e.g. temporary network disconnections are handled as part of normal SDK lifecycle). Use SBAnalytics for application-level events, or register for the reachabilityDidChange notification provided by a NetworkReachabilityService to be notified on network state changes.

To migrate localizedStringForKey

The function was moved to SBSDKConfiguration, its signature didn’t change:

    SBSDKConfiguration.localizedString(forKey: "...")

Note that if you used a SBLocalizedString or SBLocalizedPluralString functions, no change is required: the functions are still fully supported.

Target Application Project Changes

The SDK no longer supports iOS 10.x, and 32-bit architectures. To use this version of the SDK, please update your project’s Build Settings as follows:

  • Set iOS Deployment Target to iOS 11.0 or higher.
  • Ensure Valid Architectures only includes 64-bit architectures (arm64 and arm64e)

Module Usage Changes

The SDK is modifying the default location of the module map in the dynamic framework in order to accomodate an additional module map for an embedded static library. This may result in a compilation error as the compiler won’t be able to find the required module map at the default location. To use this version of the SDK, please update your project’s Build Settings as follows if applicable:

In Swift

  • If the SDK installation and integration is done via a Podfile,

    • No manual build settings changes are required . The SDK’s Podspec automatically applies all of the required build settings to the host application target via Cocoapods.

  • If the SDK is integrated via a Direct Download of the framework that is dragged and dropped into the application target’s Frameworks, Libraries, and Embedded Content section:

    • To resolve the compilation error that appears when using import SensibillUI, the application target’s build settings need to be updated. The Swift Compiler - Search Path section and Import Path setting need to be updated with the path of the SensibillUI framework with the search mechanism set to recursive.
    $(SRCROOT)/<relative_path_to>/SensibillUI.framework

In Objective-C

  • Irrespective of whether the installation and integration is done via Podfile or via Direct Download followed by a Drag and Drop in the Embed section of the host application target, the following changes must be performed:

    • Use #import <SensibillUI/SensibillUI.h> instead of the @import SensibillUI; directive.
    • Specify #import <SensibillUI/SensibillUI-Swift.h> in order to use SDK types that are prefixed with SBL. Types which starts with SBL are internally implemented in swift. In order to expose those symbols to the Objective-C compiler, the *-Swift.h header file needs to be imported.

Note: @import SensibillUI; directive is not available due to internal changes to the SDK’s module map.

For additional assistance, make sure the necessary steps are implemented as specified in the Installation guide, or refer our Installation Troubleshooting guide.

Capture Changes

Capture Initialization Changes

The SensibillCaptureNavigationController was replaced with the CaptureFlowCoordinator (SBLCaptureFlowCoordinator in Objective-C). The coordinator is responsible for initiating a new receipt capture workflow, presentation and dismissal of the Receipt Capture screen, the Receipt Expense Type Selection screen (if enabled), the Receipt Processing screen, and the Receipt Validation screens.

The following outlines the differences between the two approaches from an integrator perspective. For detailed integration instructions, please see the Receipt Capture section on the Entry Points page.

  • Previously the SensibillCaptureNavigationController was presented by a host UIViewController. Now the presentation of all screens is handled by a coordinator. A host view controller must be provided as a parameter on initialization of the CaptureFlowCoordinator (using CaptureFlowCoordinator(host:) initializer).

  • When the SensibillCaptureNavigationController was presented on a host UIViewController, it did not require a presenting class to retain its instance. Now, when using the CaptureFlowCoordinator, the created instance of the coordinator needs to be retained by a host UIViewController until the receipt capture flow (and optionally receipt processing) is finished.

  • The init?(transaction: SBLTransaction?) initializer of the SensibillCaptureNavigationController is replaced with the start(transaction:) method of the CaptureFlowCoordinator class. This method can be used for Transaction Linking while capturing a receipt.

  • The init?(transaction: SBLTransaction?, with expenseType: ReceiptExpenseType) initializer of the SensibillCaptureNavigationController was removed. Instead, update a transaction’s expenseType (with the value from ReceiptExpenseType) before providing it to the coordinator, and use the start(transaction:) method on CaptureFlowCoordinator similarly to the previous case. Example:

    ...
let coordinator = CaptureFlowCoordinator(host: self)
...
let sensibillTransaction = Transaction(externalAccountTransactionID: externalAccountTransactionID)
sensibillTransaction.sharedMetaData?.expenseType = expenseType.code
...
coordinator.start(transaction: sensibillTransaction)
...

  • The init?(transaction: SBLTransaction?, exitAfterCapture shouldExitAfterCapture: Bool) initializer of SensibillCaptureNavigationController was removed. Instead, set the exitAfterCapture property of the coordinator to true , and use the start(transaction:) method on an instance of CaptureFlowCoordinator similar to the previous case. Example:

let coordinator = CaptureFlowCoordinator(host: self)
coordinator.exitAfterCapture = true
...
let sensibillTransaction = Transaction(externalAccountTransactionID: externalAccountTransactionID)
...
coordinator.start(transaction: sensibillTransaction)
  • Use start(expenseType:) method on instance of CaptureFlowCoordinator to associate the specific expense type with the receipt after capturing. If this method is used, the Receipt Expense Type Selection screen will not be shown, and a user will be taken directly to the Receipt Processing screen.

Capture Delegate Changes

The delegate method

func sensibillCaptureNavigationController(_ viewController: SensibillCaptureNavigationController, didCreateReceipt receipt: SBReceipt?)` of `SensibillCaptureNavigationControllerDelegate

was replaced with an optional delegate method in CaptureFlowCoordinatorDelegate:

func coordinator(_ coordinator: CaptureFlowCoordinator, didProcessReceipt receipt: SBReceipt)

Note: This event will only be triggered, if user waits on Receipt Processing screen, and a receipt is successfully created. To be notified on any other outcomes, please register a transaction observer with SBDataEvent, as described in the Monitoring Receipt Uploading Status section.

Capture Dismissal Changes

Previously, if the exitAfterCapture flag was set to false, the SensibillCaptureNavigationController required a dismissDelegate of type SBViewControllerDelegate to receive an event dismissSensibillViewController(_ controller: UIViewController!) upon capture completion to be able to dismiss capture.

The SBViewControllerDelegate type is no longer used for new capture: the CaptureFlowCoordinator internally manages presentation and dismissal of all view controllers.

Instead, implement CaptureFlowCoordinatorDelegate to receive the following events:

  • The coordinatorDidCancelCapture event will notify the delegate when the user cancels a receipt capture flow by pressing a Cancel button.

  • The coordinatorDidFinishCapture event will notify the delegate when the user finishes capturing a receipt, and presses the Done button. Note: If the capture flow was started with the exitAfterCapture flag set to true, the event will be delivered to the delegate as soon as the transaction is submitted to the TransactionHandler for processing.

In both cases, the coordinator will handle the dismissal of the capture flow internally on the host view controller.

  • Optionally the delegate can also be notified when an image was submitted for processing to the server (didSubmitTransaction event), and when a resulting receipt is avalable in local user’s cache (didProcessReceipt event).

See CaptureFlowCoordinatorDelegate for detailed information on all available events.

Transaction Linking Changes

The Transaction Linking page provides the updated instructions on how to create a link between a particular receipt and a bank transaction using the new capture flow.

Capture Configuration Changes

The capture configuration is encapsulated in SBCaptureViewConfiguration, which outlines feature flags and UI properties used by both legacy and new capture, as well as feature flags and UI properties specific to legacy and new capture. Please review SBCaptureViewConfiguration for detailed information.

Receipt Uploading Changes

  • TransactionHandler will now return a new error code TransactionError.uploadCancelledImageTooBig if the imageData in the provided Transaction object is larger than 2 MB.
  • Transaction has a new initializer init(uiImage: UIImage), which allows to initialize a transaction with a provided UIImage. The image is automatically downsampled to 900px on a shorter side, converted to JPEG format, and compressed to 2 MB.
  • Transaction also has a new function updateImageData(using: UIImage), which allows you to retrieve imageData from a provided UIImage. Similar to the initializer, the image is automatically down-sampled to 900px on the shorter side, converted to JPEG format, and compressed to 2 MB.

Upgrading from 1.23 to 1.24

Summary of Changes

Integration must be updated to reflect the following changes:

  • Support for iOS 9.x has been removed. To use this version of SDK, please update your project’s iOS Deployment Target to iOS 10.0 or higher.

  • Receipt processing was changed. This change affects usage of following classes:

    • SBTransaction read more in Transaction Data Model section.
    • SBTransactionDelegate read more in Transaction Delegate section.
    • SBTransactionHandler read more in Transaction Handler section.
    • SBTransactionStatus read more in Transaction Status section.
    • SBTransactionType read more in Transaction Type section.

  • Inbox Sync feature and associated APIs are now deprecated.

SBDataEvent Changes

SBDataEventType was replaced by SBReceiptEventType for receipt observers, and SBTransactionEventType for transaction observers. The values are changed as follows:

  • SBDataEventTypeStatusUpdate event only exists for transaction observers (it was never sent for receipt observers in the past). Hence it was moved to SBTransactionEventType as .statusUpdate (SBTransactionEventTypeStatusUpdate in Objective-C)

  • SBDataEventTypeEdit event only exists for receipt observers (was never sent for transaction observers). Hence it was moved to SBReceiptEventType as .edit (SBReceiptEventTypeEdit in Objective-C)

  • SBDataEventTypeDeletion had a different meaning for receipt and transaction observers, and hence was added to both, SBReceiptEventType and SBTransactionEventType.

    • For receipt, this event is sent on receipt deletion, and is hence is called .deletion (SBReceiptEventTypeDeletion in Objective-C).
    • For transaction, the event is sent when transaction is removed from the list of transactions processed by TransactionHandler, and hence is called .removed (SBTransactionEventTypeRemoved in Objective-C)

  • Additionally, an event .expectedProcessingTimeDidChange ( SBTransactionEventTypeExpectedProcessingTimeDidChange in Objective-C) was added to SBTransactionEventType to indicate when expected processing time for a transaction have changed.

Receipt Processing Changes

Transaction Data Model

  • SBTransaction data model was replaced with the Transaction (SBLTransaction in Objective-C). Change all SBTransaction references to Transaction/SBLTransaction and review changes in fields below.

  • Every transaction is now guaranteed to have a unique non-null unchangeable localID for the entire lifetime of a transaction.

  • Previously transaction’s remoteID and receiptID had nonnull property attributes and were set to an empty value until a real value was received from the server. Now both fields are set to nil initially, and their value changes only when a corresponding value is received from the server.

For example a verification

if !transaction.receiptID.isEmpty {
  // Store transaction.receiptID
}

can either be replaced with if transaction.receiptID != nil or unwrapped as follows:

if let receiptID = transaction.receiptID {
    // Store receiptID
}

  • The status property of the new Transaction model uses a new TransactionStatus enum (SBLTransactionStatus in Objective-C). See detailed migration instructions in the Transaction Status section below.

  • The type peroperty of the new Transaction model uses a new TransactionType enum (SBLTransactionType in Objective-C). See detailed migration instructions in the Transaction Type section below.

  • Transaction now has a new error field, containing a value from TransactionError enum. When transaction status is set to failed, error field will contain the reason of failure as error code. Otherwise error field will be set to noError code.

  • The delegate which implemented SBTransactionDelegate interface in the SBTransaction is now changed to observer, and implements TransactionObserver interface. See Transaction Delegate section below.

  • Initializer Transaction(imageData: Data) no longer allows initialization with the nil image (imageData is not optional). For usage with Transaction Linking, the new initializer is provided: Transaction((externalAccountTransactionID: String).

Replace the following lines:

let uploadTransaction = SBTransaction(imageData: nil)
uploadTransaction.privateMetaData?.externalAccountTransactionID = "transactionID"

with

let uploadTransaction = Transaction(externalAccountTransactionID: "transactionID")

Transaction Status

SBTransactionStatus was replaced with the TransactionStatus, which reflects the lifecycle of Transaction.

  • Previous status needsSynchronization should be replaced with the uploading status. This status indicates, that receipt was saved locally, but before it was uploaded to the server, or server accepted it for processing. This status is followed with the processing, or failed.

  • Statuses pending and processing previously had the same value, and could be used interchargeably. Now status pending is removed. Status processing is used to indicate that server accepted receipt for processing.

  • Transaction remains in the processing status until the server and SDK fully complete their processing. This status will change to completed (if the receipt is processed successfully), or failed (if an error is received).

    Note that if the server identifies a duplicate transaction, which was already successfully processed in the past, the SDK will set the completed status immediately, the transaction’s receiptID will be set to an existing receipt ID, and the transactionID field will remain empty.

  • Statuses uploaded and completed had the same value, and could be used interchangeably. Now the status uploaded was removed. Use completed instead.

  • Status stopped was removed. Previously this status was set when the server returned a status stopped. Now in the same case, the status will be set to failed, and the error property will be set to wasStoppedOnServer. For example, the following condition: transaction.status == SBLTransactionStatusFailed || transaction.status == SBLTransactionStatusStopped should be changed to: transaction.status == SBLTransactionStatusFailed In cases where it’s important to differentiate stopped status returned by the server, change transaction.status == SBLTransactionStatusStopped with transaction.error == SBLTransactionErrorWasStoppedOnServer

The following values have the same name and meaning as the previous version:

  • unknown
  • completed
  • failed

Transaction Delegate

SBTransactionDelegate was replaced with the TransactionObserver (SBLTransactionObserver in Objective-C).

  • Change protocol name from SBTransactionDelegate to TransactionObserver in Swift (SBLTransactionObserver in Objective-C).

  • Function statusDidChange(for transaction: SBTransaction) should be changed to statusDidChangeForTransaction(_ transaction: Transaction) in Swift (statusDidChangeForTransaction:(SBLTransaction * _Nonnull)transaction in Objective-C)

  • Function expectedProcessingTimeDidChange(for transaction: SBTransaction) should be changed to expectedProcessingTimeDidChangeForTransaction(_ transaction: Transaction)

Transaction Handler

  • SBTransactionHandler was replaced with the TransactionHandler (SBLTransactionHandler in Objective-C).

  • Previously if an integrated application was accessing SBTransactionHandler.sharedHandler(), an explicit cast from the Any type was required in Swift. Now, a shared instance of TransactionHandler can be accessed with the TransactionHandler.shared, and no casting is required.

  • addTransaction function was changed to add

  • removeTransaction function was changed to remove

  • removeProcessedEmailTransaction function was removed. Instead, find the transaction with the receiptID you want to remove, and pass it to the remove function:

guard let transaction = TransactionHandler.shared.transaction(forReceiptID: "receiptID") else {
    // Not found, nothing to remove
    return
}
TransactionHandler.shared.remove(transaction)
  • Filters allFailedImageTransactions, allFailedEmailTransactions, allProcessingTransactions, and allOffineTransactions were changed to properties failedImageTransactions, failedEmailTransactions, processingTransactions, and offineTransactions respectively.
  • Filter allNeedsSynchronizationTransactions was changed to inFlightTransactions variable to clarify its meaning. This filter contains all functions that have not yet complete (i.e. status is not failed or completed)

  • Filter named allProcessedEmailTransactions was changed to completedEmailTransactions to reflect its status.

  • Function convertEmailToReceipt was migrated as convert with similar arguments.

  • Function - (void)transaction:(SBLTransaction * _Nonnull)transaction changeClassificationTo:(SBLTransactionType)type success:(void (^ __nullable)(BOOL didSucceed))success is no longer required. Instead, create a transaction instance using Transaction(forNonReceiptEmail: SBNonReceiptEmail) initializer directly, and then add transaction for processing.

  • Access to monitoring and persistence functions (monitorTransactionAtSoonestDate, synchronizeUpdatedTransaction, synchronizeTransactionQueueFromDisk, synchronizeTransactionQueueToDisk) is no longer available for external callers, and are not necessary. TransactionHandler now continiously verifies which transactions should be uploaded, based on network and application state, transaction status, and transaction retry policy. Transaction Notifications

Notification types kReceiptUploadCompleted, kReceiptUploadCompletedWithReceiptId, kReceiptUploadCompletedWithoutExpenseType and kReceiptUploadCompletedWithoutExpenseTypeWithReceiptId are no longer sent by SDK.     let configuration = SBSDKConfiguration.sharedInstance()

    if transaction.receiptID != nil {

        // Matches previous kReceiptUploadCompleted case
        if withoutExpenseType {
            // Matches previous kReceiptUploadCompletedWithoutExpenseTypeWithReceiptId case
        } else {
         // Matches previous kReceiptUploadCompletedWithReceiptId case
        }
    } else {

        if withoutExpenseType {
            // Matches previous kReceiptUploadCompletedWithoutExpenseType case
        } else {
            // Matches previous kReceiptUploadCompleted case
        }
   }
}

Notification kReceiptProcessingBegan corresponds to a status change notification, when the status of a transaction is set to uploading:

func statusDidChangeForTransaction(_ transaction: Transaction) {
    if transaction.status == .uploading {
        // Matches previous kReceiptProcessingBegan case
    }
}

Similarly, notification kReceiptProcessingEnded corresponds to a status change notification, when the status of a transaction is set to completed or failed:

func statusDidChangeForTransaction(_ transaction: Transaction) {
    if transaction.status == .completed || transaction.status == .failed {
        // Matches previous kReceiptProcessingEnded case
    }
}

Alternatively you can check the isFinal flag of the status. This flag is set to true only for completed and failed statuses.

func statusDidChangeForTransaction(_ transaction: Transaction) {
    if transaction.status.isFinal {
        // Matches previous kReceiptProcessingEnded case
    }
}

Notification kReceiptProcessingExpectedTimeUpdated is no longer sent by SDK. Instead a caller can use TransactionObserver to monitor the status of a particular transaction (expectedProcessingTimeDidChangeForTransaction event), or add itself as a transaction observer in the SBDataEvent to monitor the status changes for all transactions (SBTransactionEventTypeExpectedProcessingTimeDidChange event type).

Starting and Stopping SDK

To start the SDK, instead of Authenticator(tokenProvider).start(cacheIdentifier:completion), the integrator must call

Sensibill.start(tokenProvider:cacheIdentifier:completion)

where

  • the tokenProvider argument is the same as what was previously passed to the Authenticator instance.
  • the cacheIdentifier argument is the same as the identifier argument from the obsolete implementation of Authenticator’s start function
  • the completion callback now returns a SensibillError instead of an Error interface. See below for more details.

To stop the SDK and invalidate a current user session, instead of using PasswordAuthenticator().logout(completion), the integrator must call

Sensibill.stop(completion)

The completion callback for both start and stop now returns a SensibillError instead of an Error interface, as specified in Summary of Changes.

Additionally, the SDK now allows start to be called only after the SDK is stopped. For more information and example of the new start and stop implementation, see SDK Lifecycle

The PasswordAuthenticator

The start function inside PasswordAuthenticator is now deprecated and should not be used.

  • To start the SDK with a custom TokenProvider, use the Sensibill.start function instead as explained in the Starting and Stopping SDK section above
  • To use password authentication, use PasswordAuthenticator’s login function to start the SDK.

Similarly to Sensibill.start and Sensibill.stop:

  • all functions in PasswordAuthenticator with a completion callback now return a SensibillError instead of an Error interface. Old generic errors have been replaced with more useful errors that will now return the reason for failure.
  • the login function can now only be called after an existing user session is invalidated using the logout function or Sensibill.stop.

See Password Authentication for more information and examples of the completion callback implementation.

Note: mixing authentication types or changing token providers within the same user session is no longer supported. The integrator must call Sensibill.stop to use a different authentication type, or use a different token provider.
