Configuration

Branding Configuration

There are two layers to branding configuration. The easiest way for most purposes is to use the colors property of the SBSDKConfiguration singleton object. The properties of colors will be used by the SDK to map your branding to all visual elements of the SDK. When the SBSDKConfiguration singleton is first accessed, it will be initialized with a default set of values for the colors of the SDK, but you can override these before starting the SDK:

Swift

let configuration = SBSDKConfiguration.sharedInstance()

configuration?.colors.brandBackground = .red
configuration?.colors.brandForeground = .blue
configuration?.colors.brandForegroundDisabled = .gray
configuration?.colors.actionBackground = .purple
configuration?.colors.actionForeground = .orange

configuration?.applyColors()

Objective-C

SBSDKConfiguration *configuration = [SBSDKConfiguration sharedInstance];

configuration.colors.brandBackground = [UIColor redColor];
configuration.colors.brandForeground = [UIColor blueColor];
configuration.colors.brandForegroundDisabled = [UIColor greyColor];
configuration.colors.actionBackground = [UIColor purpleColor];
configuration.colors.actionForeground = [UIColor orangeColor];

[configuration applyColors];

There are plenty of other colors one can set, but these are just some examples. Using the small number of properties on the color object, you can quickly brand the SDK with minimal work.

Since you are setting these properties on the singleton configuration object, there are no other changes needed (i.e. don’t need to pass this object anywhere) as any UI which is created in the SDK next will use the new values. All you need to do is call applyColors when you are done setting your custom colors on the SBSDKConfiguration singleton and these colors will be mapped to their respective settings for you.

The second way of setting these values (which could be used together with the first method) is to override the underlying properties which those initial properties map to. The only reason you would want to use this method was if you wanted an individual element on one screen to appear differently than a similar element on another screen. Usually these two elements would have the same appearance (they’d both use the child property of colors, as shown above), but in this case you may want to have more granularity:

Swift

let configuration = SBSDKConfiguration.sharedInstance()

configuration?.loadingViewConfiguration.loadingBackgroundColor = .black
configuration?.loadingViewConfiguration.loadingSpinnerColor = .white

Objective-C

SBSDKConfiguration *configuration = [SBSDKConfiguration sharedInstance];

configuration.loadingViewConfiguration.loadingBackgroundColor = [UIColor blackColor];
configuration.loadingViewConfiguration.loadingSpinnerColor = [UIColor whiteColor];

Above you can see a simple example for overriding the colors of the loading view. If you didn’t override the colors like this, they’d take on the colors defined in the higher-level configurations (in this example, brandBackground & brandForeground).

This section just outlines the basic way of changing color configurations, but there is more extensive documentation on all the configuration options available in another document, so please refer to the iOS Customization Document.

Font Configuration

To customize the font in the Sensibill SDK, the SBSDKConfiguration instance provides a way to update the fonts in the SDK UI.

  1. Get the shared instance of the SBSDKConfiguration that the SDK reads from by calling SBSDKConfiguration.sharedInstance()

  2. Access the shared instance font configuration by calling SBSDKConfiguration.sharedInstance().font

  3. To change the font, adjust the properties of the font configuration in the shared instance. To change the bold font, assign the desired bold font to SBSDKConfiguration.sharedInstance().font.boldFont. To change the regular font, assign the desired font to SBSDKConfiguration.sharedInstance().font.regularFont

Asset Configuration

To customize assets in the Sensibill SDK, an external Bundle must be added to the SDK first. The SDK will use this added bundle to check for assets to use in the Sensibill UI. The assets inside of the Bundle must have the same name as assets already used in the SDK. By doing so, the provided assets override what is in the SDK. The SBSDKConfiguration instance provides a way to add an external bundle that the SDK will use for images and strings.

  1. Create a Bundle in Host application’s project, called SensibillOverride

  2. Inside the Bundle created in Step 1, create an Asset Catalog called SensibillAssets.xcassets

  3. Get the shared instance of the SBSDKConfiguration by calling SBSDKConfiguration.sharedInstance()

  4. Create a Bundle instance, pointing to the path of the SensibillOverride bundle created in Step 1

  5. Add the Bundle into to the shared instance of SBSDKConfiguration by assigning value of externalBundle to Bundle instance created in Step 4

let sensibillConfig = SBSDKConfiguration.sharedInstance()

if let bundleURL = Bundle.main.url(forResource: "SensibillOverride", withExtension: "bundle") {
    let bundle = Bundle(url: bundleURL)
    sensibillConfig?.externalBundle = bundle
}

Image Configuration

Once you have provided an external bundle that the Sensibill SDK will use to retrieve assets, images in the SDK can be overriden.

Make sure to follow the steps in Asset Configuration first

  1. Prepare the desired imageset and rename to match the imageset you want to replaced. Example name imageset to capture-close to override close icon in Receipt capture

  2. Copy imageset from step 1 into the Asset Catalog SensibillAssets.xcassets from the Asset Configuration steps

String Configuration

Similar to Image configuration, strings in SDK can be customized by adding LocalizableStrings containing the strings to be used to the external bundle.

Make sure to follow the steps in Asset Configuration first

  1. Prepare LocalizableStrings with strings to override. Key of string should match key of string in SDK to override. Value of string should be the content you want to override with

  2. To support multiple languages, various of LocalizableStrings files matching region/language you want to customize for can be added to Bundle.
